# Installation Guide

InsightOCRv2 ships as a Docker Compose stack — 8 services (backend, frontend, db, redis, minio, nginx, celery_worker, celery_beat) that run on a single host.

## Quick start

```bash
git clone <repo-url> insightocr
cd insightocr
./scripts/setup.sh
```

When setup completes, open `http://localhost` in your browser and log in with `admin@softnix.ai`.

That's it. The rest of this document explains what `setup.sh` does, OS-specific notes, troubleshooting, and how to harden the install for production.

---

## Prerequisites

| Requirement | Minimum | Notes |
|---|---|---|
| Docker Engine | 24.0+ | [Install instructions](https://docs.docker.com/engine/install/) |
| Docker Compose | v2 (plugin) | v1 (`docker-compose`) works with a warning |
| RAM | 8 GB | backend 2G + celery 3G + db/minio/nginx ~3G |
| Free disk | 10 GB | Images ~3 GB + document uploads + sandbox layer cache |
| OS | macOS / Linux / WSL2 | Native Windows not supported (use WSL2) |

You don't need to install anything else — Python, Node.js, PostgreSQL, Redis, MinIO are all containerized.

---

## What `setup.sh` does

Six phases. All are idempotent — safe to re-run.

### A. Prerequisites check (~5s)
Detects OS, verifies Docker is installed and running, checks RAM ≥ 8 GB (warns only), disk ≥ 10 GB (blocks), and that `/var/run/docker.sock` is accessible. On macOS it tries to start Docker Desktop automatically if the daemon isn't running.

### B. Secrets + env files (~1s)
If `.env` and `backend/.env` don't already exist, copies them from the `.example` templates and replaces every `CHANGE_ME_WILL_BE_GENERATED` sentinel with `openssl rand -hex`. Existing `.env` files are never touched unless you pass `--reset-secrets`.

### C. OCR endpoint configuration (REQUIRED)
External OCR is required for document processing. If `OCR_ENDPOINT` is empty in `backend/.env`, the script pauses and prompts you to paste an endpoint (and optional API token). Press ENTER to skip — services will still start, but document upload/processing won't work until you set it. Configure later via `backend/.env` or the `/settings` page.

### D. Build + start (~3-5 min first run, ~30s subsequent)
Runs `docker compose build --pull` then `docker compose up -d`. First run downloads base images (Python 3.11, Node 20, Postgres 15, etc.) and builds the sandbox image — expect ~3 min. Subsequent runs only rebuild if `Dockerfile`s changed.

### E. Health checks (~30s)
Polls `db`, `redis`, `backend`, `frontend` health endpoints. Prints `✓` per service or `✗` with timeout. 5-minute timeout per service, configurable via `--health-timeout`.

### F. Summary
Prints URLs, admin credentials, and next steps.

## OS-specific notes

### macOS
- Docker Desktop: allocate at least 8 GB RAM in **Settings → Resources**.
- `setup.sh` auto-starts Docker Desktop if needed.
- Apple Silicon (M1/M2/M3) works — `platform: linux/amd64` is set in compose where needed.

### Linux
- Add your user to the `docker` group **before** running setup:
  ```bash
  sudo usermod -aG docker $USER
  newgrp docker  # or log out + log back in
  ```
  Without this, the script will exit with instructions — auto-fixing group membership would require sudo + re-login, which we don't do automatically.
- Docker socket path: `/var/run/docker.sock` (default).

### WSL2 (Windows)
- Install **Docker Desktop for Windows** with WSL2 integration enabled.
- Run `setup.sh` from inside your WSL2 distro (Ubuntu, Debian, etc.).
- Docker Desktop auto-starts when you call `docker` from WSL — usually no manual start needed.

## Configuration

### Environment files
| File | Purpose |
|---|---|
| `.env` | Root compose credentials — DB/Redis/MinIO passwords, SECRET_KEY |
| `backend/.env` | Backend app config — OCR endpoint, storage, AI provider |

Both are generated by `setup.sh` and **should not be committed** (already in `.gitignore`).

### Changing secrets later
```bash
./scripts/setup.sh --reset-secrets   # regenerates everything (DESTRUCTIVE)
```

Then restart:
```bash
docker compose down
docker compose up -d
```

### Switching storage backend
Default is `local` (docker volume). To use MinIO or AWS S3, edit `STORAGE_TYPE` + related vars in `backend/.env` and restart backend.

## Verification

After `setup.sh` prints "✓ ready":

1. **Web UI**: open `http://localhost` — should see login page
2. **Login**: `admin@softnix.ai` / `admin` (change immediately in `/profile`)
3. **API docs**: `http://localhost/api/v1/docs` — interactive Swagger UI
4. **Upload smoke test**: create a job, upload a small PDF, click **Process**
5. **Agent**: open AI Agent tab in job detail — should see empty state with suggested prompts

If any step fails, see Troubleshooting below.

## Troubleshooting

### Port 80 / 443 already in use
```bash
sudo lsof -i :80 -i :443    # find what's holding the port
docker compose down          # stop our stack
# stop the conflicting service, then:
docker compose up -d
```

Or run on a different port — edit `nginx/conf.d/default.conf` `listen 80` → `listen 8080`.

### Sandbox image build fails
The `Dockerfile.sandbox` builds `insightocr-sandbox:py312`. If it fails:
```bash
docker compose build celery_worker 2>&1 | tail -20
docker build -f backend/Dockerfile.sandbox -t insightocr-sandbox:py312 backend/
```

The most common cause is DNS resolution during `pip install`. Try:
```bash
docker build --network=host -f backend/Dockerfile.sandbox -t insightocr-sandbox:py312 backend/
```

### Backend not healthy
```bash
docker compose logs backend | tail -50
```
Most common causes:
- Missing `SECRET_KEY` in root `.env` (re-run `setup.sh`)
- DB not ready yet (wait 30s and check again)
- OCR endpoint unreachable (warning only, shouldn't block startup)

### Document processing fails
- Check `OCR_ENDPOINT` and `API_TOKEN` in `backend/.env`
- Test connectivity: `curl -sf $OCR_ENDPOINT/../me` (drop `/ai-process-file` suffix)
- Check `docker compose logs celery_worker`

### Reset everything (DESTRUCTIVE)
```bash
docker compose down -v        # removes volumes = all data
rm -f .env backend/.env
./scripts/setup.sh
```

## Production hardening

The default compose is tuned for dev/test on localhost. For production:

1. **TLS**: replace `nginx/ssl/cert.pem` + `key.pem` with real certs (Let's Encrypt or your CA). See `nginx/ssl/generate-certs.sh` for dev-only self-signed.
2. **Secrets**: never use `--reset-secrets` on prod; rotate manually via a secrets manager.
3. **Backups**: schedule `pg_dump` for the `db` service + mirror the `documents_uploads` volume to off-host storage.
4. **Resource limits**: review `deploy.resources.limits` in `docker-compose.yml` — production may need more.
5. **External OCR**: production OCR endpoint should have its own SLA — monitor connectivity.
6. **Monitoring**: add log aggregation (Loki/Datadog) + metrics (Prometheus exporter for Postgres/Redis).
7. **Network**: consider removing `/var/run/docker.sock` mount in production (disables sandbox — alternative is remote code execution service).

## Re-running setup

All flags:

```bash
./scripts/setup.sh                    # standard fresh run
./scripts/setup.sh --skip-ocr-prompt  # don't pause for OCR config
./scripts/setup.sh --no-build         # use existing images
./scripts/setup.sh --reset-secrets    # regenerate secrets (DESTRUCTIVE)
./scripts/setup.sh --health-timeout 600  # 10-min health timeout per service
./scripts/setup.sh -h                 # show help
```

The script is idempotent — running it again skips already-completed phases.
