# Manual Test Checklist: Security & Storage Updates

เอกสารนี้รวบรวมรายการทดสอบ (Checklist) สำหรับตรวจสอบความถูกต้องของระบบหลังจากมีการแก้ไขช่องโหว่ความปลอดภัยและการเพิ่มระบบ Storage แบบใหม่

| 1. ชื่อ Feature / หัวข้อ | 2. สิ่งที่แก้ไข / เปลี่ยนแปลง | 3. ผลลัพธ์ที่ควรจะเป็น (Expected Result) | 4. แนวทางการทดสอบ (Test Steps) |
| :--- | :--- | :--- | :--- |
| **1. Secret Key Security** | ย้าย `SECRET_KEY` จากใน Code (`config.py`) ไปไว้ใน `.env` และบังคับให้อ่านจาก Env เท่านั้น | ระบบทำงานได้ปกติ ไม่มีการ Crash และ Token เก่ายังใช้งานได้ (ถ้าใช้ Key เดิม) หรือต้อง Login ใหม่ (ถ้า Key เปลี่ยน) | 1. ตรวจสอบไฟล์ `.env` ต้องมี `SECRET_KEY`<br>2. ลอง Login เข้าใช้งานระบบ<br>3. ถ้าลบ `SECRET_KEY` ออกจาก `.env` แล้ว Restart, Server ต้องเริ่มไม่ติดหรือแจ้งเตือน |
| **2. API Authentication** | เพิ่มการตรวจสอบ Token ในทุก API ที่สำคัญ (Upload, Job List, Download) | User ที่ยังไม่ Login (ไม่มี Token) ต้องไม่สามารถเรียก API ได้ (Error 401 Unauthorized) | 1. ใช้ Postman/Curl เรียก `GET /api/v1/jobs/` โดย **ไม่ใส่** Header Authenticaion -> ต้องได้ 401<br>2. Login แล้วนำ Token มาใส่ Header -> ต้องได้ 200 OK |
| **3. Job Data Privacy (RBAC)** | จำกัดสิทธิ์การมองเห็น Job (`jobs.py`). User ธรรมดาเห็นแค่ของตัวเอง, Admin เห็นทั้งหมด | User A ต้องไม่เห็น Job ของ User B | 1. Login ด้วย User A สร้าง Job 1 งาน<br>2. Login ด้วย User B แล้วเรียกดู List Jobs -> **ต้องไม่เห็น** Job ของ User A<br>3. Login ด้วย Admin -> ต้องเห็น Job ของทั้ง A และ B |
| **4. CORS Security** | ลบ `allow_origin_regex=".*"` ที่อนุญาตทุกโดเมน ออกจาก `main.py` | เว็บไซต์อื่นที่ไม่ใช่ Frontend ของเรา (Localhost 3000) ต้องเรียก API ไม่ได้ | 1. เปิด Console ใน Browser บนเว็บอื่น (เช่น Google)<br>2. สั่ง `fetch('http://localhost:8000/health')`<br>3. ต้องเจอ Error CORS Blocked (Red text in console) |
| **5. Path Traversal Protection** | เพิ่ม Logic ใน API Download ตรวจสอบว่าไฟล์ต้องอยู่ในโฟลเดอร์ `/app/uploads` เท่านั้น | ไม่สามารถ Download ไฟล์ระบบ (เช่น `/etc/passwd`) ได้ | 1. (Advanced) ลองแก้ Database ในตาราง `document` เปลี่ยน `file_path` เป็น `/etc/passwd`<br>2. เรียก API Download Document ID นั้น<br>3. ต้องได้ Error 400 หรือ 404 (ไม่โหลดไฟล์ออกมา) |
| **6. Storage Abstraction (Default)** | เปลี่ยน Code ให้ใช้ `StorageService` แต่ Config เป็น `STORAGE_TYPE=local` | ระบบ Upload/Download/Process OCR ทำงานได้ปกติเหมือนเดิมทุกอย่าง | 1. Upload ไฟล์เอกสารใหม่<br>2. รอ OCR Process จนเสร็จ<br>3. ลองกด Download ไฟล์กลับมา<br>4. ตรวจสอบว่าไฟล์ยังถูกสร้างในโฟลเดอร์ `backend/uploads` |
| **7. MinIO Storage Support** | เปลี่ยน Config `.env` เป็น `STORAGE_TYPE=minio` และตั้งค่า MinIO | ไฟล์ที่ Upload ใหม่ต้องไปโผล่ใน MinIO Bucket แทนที่จะอยู่ในโฟลเดอร์ local | 1. แก้ `.env` เป็น `minio`, `docker compose up minio`<br>2. Upload ไฟล์ใหม่<br>3. เข้าหน้า Web Console MinIO (Port 9001) เช็คว่ามีไฟล์เข้า Bucket `insightocr`<br>4. หน้าเว็บต้อง Download ไฟล์และแสดงผล OCR ได้ปกติ |
| **8. S3 Storage Support (Optional)** | เปลี่ยน Config `.env` เป็น `STORAGE_TYPE=s3` และใส่ Credential AWS จริง | ไฟล์ที่ Upload ใหม่ต้องไปโผล่ใน AWS S3 | (ถ้ามี AWS Account) <br>1. แก้ `.env` ใส่ Key AWS<br>2. Upload ไฟล์ -> เช็คใน AWS Console S3 Bucket |
